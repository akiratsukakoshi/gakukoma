## 2. 課題詳細と根本原因分析

### A. レスポンス速度（現状約15秒）

#### 推定時間内訳

| フェーズ | 推定時間 |
|---|---|
| VAD + STT (Whisper small) | 約2〜3秒 |
| OpenClaw APIコール（LLM） | 約10〜12秒 |
| TTS (Open JTalk) | 約0.1秒 |

**ボトルネックはLLMコール**。会話が積み重なるほどOpenClawが全セッション履歴をAPIに送るため、入力トークンが増加しレイテンシが増大する。

#### A-1. ローリングウィンドウ + コンパクション

**方針**:
- 直近N턴（例: 5〜8往復）の会話履歴のみをフル形式で保持
- 古い履歴はLLMに要約させ「コンパクション済みメモリブロック」として先頭に置く

```
[システムプロンプト（キャッシュ対象）]
[コンパクション済みメモリ: 「以前〇〇について話した...」]
[直近5〜8ターンの生の会話履歴]
```

- 要約タイミング: N턴超過時に非同期で実施（レスポンスをブロックしない）
- メモリブロックはセッション間を超えて保持可能にする（長期記憶）

#### A-2. プロンプトキャッシュ

**仕様（Anthropic Prompt Caching）**:

| 項目 | 内容 |
|---|---|
| 対応モデル | Claude Haiku 3.5 ✅（使用中） |
| キャッシュ有効時間 | 5分（ephemeral） |
| 読み込みコスト | 通常の10%（90%削減） |
| 最小トークン数 | 1,024トークン（Haiku） |
| 設定方法 | `cache_control: {type: "ephemeral"}` をメッセージに付与 |

**適用箇所**: システムプロンプト（SOUL.md + TOOLS.md等）をキャッシュ対象にする。
会話履歴（動的）はキャッシュ外。

**制約**: TTL5分のため、ウェイクワード待機中（無音状態）が5分超えるとキャッシュ失効。
→ まず実装して実測し、失効が問題になるか確認してから対処方針を決める。

#### 削減効果試算（A-1 + A-2 合算）

| 施策 | 効果 |
|---|---|
| ローリングウィンドウ | 会話が長くなっても入力トークンを一定に抑制 |
| プロンプトキャッシュ | 静的プロンプト部分のレイテンシ削減（推定1〜2秒） |
| 合算目標 | **15秒 → 8秒以内**（目標値） |

#### 将来検討（今回は対象外）

- **ストリーミング発話**: LLMレスポンスが来はじめた時点でTTSを開始する。体感4〜6秒削減見込みだが、Open JTalkはストリーム非対応のため文節分割処理の実装が必要。Phase 3後以降に検討。
- **顔識別**: `face_recognition`ライブラリ（dlib/aarch64）で実装可能。Pi5の負荷が高いため静止画判定方式を検討。今回は見送り。

### B. ビジュアル認識

#### B-1. see_aroundの視点問題

**症状**: 「ガクチョが自撮りをしている」「写真を見ると〜」など、第三者的な評価になっている。

**根本原因**: `see_around.sh` がVision APIに画像を渡す際のコンテキストプロンプトに、
「これはGAKUKOMA自身のカメラ映像（自分の視野）」という情報が明示されていない。

**方針**: Vision APIへのプロンプトに下記を追加する（1行の修正）。

```
この画像はあなた自身（がくこま）の前面カメラが捉えた、あなたの現在の視野です。
```

#### B-2. 部屋レイアウト認識（Phase 3向け準備）

**目標**: Phase 3で「入口・ドアを探して移動する」ためのナビゲーション情報取得。

**方針**: パン・チルトを使ったパノラマスキャン実装。

```
survey_room() 関数フロー:
1. look_direction left  → see_around（左側撮影）
2. look_center          → see_around（正面撮影）
3. look_direction right → see_around（右側撮影）
4. look_center          → 3枚の画像をVision APIに一括送信
5. プロンプト: 「ドア・入口・通路の方向を時計方向で列挙してください」
```

- 深度カメラなしでも「大まかな方向判断ナビ」は可能
- 正確な障害物距離計測はPhase 4以降（深度カメラ検討）

## 6. リスクと対策

| リスク | 対策 |
|---|---|
| OpenClawの内部履歴管理がブラックボックスで改修困難 | CLIラッパー側で履歴ファイルを独自管理する代替実装 |
| キャッシュTTL5分でウェイクワード待機中に失効 | まず実装・実測。失効が問題になれば「定期的な軽量Ping APIコール」で更新する仕組みを追加 |
| コンパクション要約のAPI呼び出しがレスポンスをブロック | 非同期で実施（要約は次のターン開始前に完了させる） |
| workspace整理後にがくこまの挙動が変わる | 整理前にgit commitでバックアップ。問題があればrevertできる状態で作業 |
| survey_room()が15〜20秒かかり体験が悪い | 「部屋を調べるね、少し待ってて」と事前に発話してから実行する |
