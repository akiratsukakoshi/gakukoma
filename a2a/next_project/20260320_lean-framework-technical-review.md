## 1. OpenClawの現行構造（把握した内容）

レビューにあたりコードベース全体を精査した。構造は以下のとおり。

```
voice_loop.py（ローカル）
  └─ call_openclaw()  ← ここだけクラウド依存
      └─ openclaw binary → Claude Haiku API
          └─ システムプロンプト: 約25,798文字（毎ターン注入）
              ├─ AGENTS.md:   7,809文字（フレームワーク運用ルール）
              ├─ TOOLS.md:    3,094文字（ツール定義）
              ├─ SOUL.md:     1,322文字（キャラクター設定）
              └─ その他: IDENTITY, USER, HEARTBEAT 等
```

**重要な前提**: STT（Whisper）・TTS（OpenJTalk）・サーボ制御（PCA9685）・カメラ（OpenCV）は**すでに全てローカル動作**している。クラウド依存は実質「LLM推論」の1点のみ。

---

## 2. 技術的実現可能性

**結論：可能。ただし「速くなる」という期待には注意が必要。**

### Pi5でのローカルLLM推論速度の現実

| モデル | 量子化 | 推定速度（Pi5 CPU） | 50トークン応答にかかる時間 |
|---|---|---|---|
| Llama 3.2 3B | Q4_K_M | 3〜6 tok/s | 8〜17秒 |
| Phi-3.5 Mini | Q4 | 4〜7 tok/s | 7〜13秒 |
| Qwen 2.5 3B | Q4 | 3〜6 tok/s | 8〜17秒 |

現在のClaude Haiku（クラウドAPI）は応答+ネットワーク込みで**2〜4秒**。
ローカル3Bモデルはむしろ**遅くなる可能性が高い**。

### 3Bモデルのツールコーリング能力

現行ではHaikuが `look_direction`, `see_around` 等を状況判断で呼び出している。3Bモデルをそのまま置き換えると、ファインチューニングなしではツール呼び出しの信頼性が大きく低下する。ファインチューニング後でも「隣の部屋に行って友人に声をかけて」のような多段推論は3Bの容量では厳しい。

## 4. 具体的な構成案

### 案A: ハイブリッド構成（現実的・まず試すならこれ）

```
ユーザー発話（STT → テキスト）
  │
  ├─ 意図分類（ローカル：軽量モデル or ルールベース）
  │   ├─ "右向いて"    → look_direction(right) を即実行（LLM不要）
  │   ├─ "何が見える？" → see_around() → Haiku（説明生成のみ）
  │   └─ 複雑な推論    → Haiku（フォールバック）
  │
  └─ 結果をTTS（OpenJTalk）で発話
```

現行ログを見ると、コマンドの大半は「向く」「見る」「動く」の単純な身体操作。これはLLMなしでもルールベースで処理できる。複雑な推論のみクラウドに回すハイブリッドが現実的。

### 案B: フルローカル化（理想形・長期目標）

```
voice_loop.py（現行ほぼそのまま流用）
  └─ call_openclaw() を call_local_llm() に置き換えるだけ
      └─ Ollama（常駐プロセス）
          └─ Qwen 2.5 3B（ファインチューニング済み）
              └─ システムプロンプト: 25KB → 2KB以下（残りはウェイトに焼き込み）
```

---

## 5. 「25KBシステムプロンプト」の解体方針

ファインチューニングで「ウェイトに焼き込める」ものと「ランタイム注入が必要なもの」を分離する。

| ファイル | 現行サイズ | ファインチューニングで焼き込める? |
|---|---|---|
| SOUL.md（キャラクター性） | 1,322文字 | ◎ 焼き込み可能 |
| TOOLS.md（ツール定義） | 3,094文字 | ○ 半分は焼き込み可能 |
| AGENTS.md（フレームワーク運用ルール） | 7,809文字 | ◎ ほぼ削除可能（フレームワーク都合のルールが大半） |
| IDENTITY.md / USER.md 等 | 数百文字 | △ セッション毎に変わる情報はランタイム注入が必要 |
| 当日のメモリ（memory/YYYY-MM-DD.md） | 可変 | × ランタイム注入が必要（動的情報） |

**削減見込み**: 25KB → 2〜3KB程度まで圧縮可能。

## 6. 学習データについて

セッションログが以下に蓄積されている。

```
~/.openclaw/agents/main/sessions/*.jsonl
```

`.jsonl` 形式でツール呼び出しの成功例が全て記録されている。これが**そのままファインチューニングの素材**になる。Phase 3（走行）・Phase 4（把持）が完了すると全身体スキルをカバーしたデータセットが揃う。そのタイミングでUnsloth on Colabを実施するのが最も効果的。

## 7. 既存資産の流用可能性

フルローカル化した場合でも、以下は**そのまま流用可能**。

| コンポーネント | 流用可否 | 備考 |
|---|---|---|
| `voice_loop.py`（メインループ） | ◎ | `call_openclaw()` を置き換えるだけ |
| STT（Whisper + VAD） | ◎ | 変更なし |
| TTS（OpenJTalk mei） | ◎ | 変更なし |
| サーボ制御（`pan_tilt.py`） | ◎ | 変更なし |
| カメラ・顔追跡 | ◎ | 変更なし |
| ツールのシェルスクリプト群（`tools/*.sh`） | ◎ | 変更なし |
| メモリ形式（MDファイル） | ◎ | シンプルで優秀な設計 |
| OpenClaw binary | × | Ollama/llama.cppで置き換え |

## 8. 推奨アプローチ（段階的移行）

```
Step 1: GAKUKOMA Phase 3・4 完了（走行・把持の実装と実績ログ蓄積）
  │
Step 2: セッションログ（.jsonl）からデータセット作成
         └─ 成功したツール呼び出しパターンを抽出・整形
  │
Step 3: Unsloth（Google Colab）でQwen 2.5 3B をファインチューニング
         └─ GGUF Q4_K_M形式で書き出し
  │
Step 4: Pi5上でOllamaセットアップ + 案Aハイブリッドで動作検証
         └─ 精度・速度を実測
  │
Step 5: 精度が十分なら案Bフルローカル化へ移行
         └─ voice_loop.py の call_openclaw() を call_local_llm() に置き換え
```
