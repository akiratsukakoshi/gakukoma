## 背景・目的

音声入力から応答までの時間が現状約15秒かかっています。
内訳の推定: STT 約2〜3秒 / **LLMコール 約10〜12秒** / TTS 約0.1秒

主因はOpenClawが全セッション履歴をAPIに毎回送信していること。
会話が積み重なるほど入力トークンが増え、レイテンシが悪化します。

**目標**: レスポンスを8秒以内に抑える。

## タスク A-1: ローリングウィンドウ + コンパクション

### 実装方針

会話履歴の保持上限を設け、古い履歴はLLMに要約させてメモリブロックとして保存します。

```
APIに送る内容（構造）:
┌─────────────────────────────────┐
│ システムプロンプト（SOUL.md等） │
├─────────────────────────────────┤
│ [コンパクション済みメモリ]      │  ← 古い会話の要約（任意・あれば）
│ 「以前、音楽について話した。    │
│  ガクチョはジャズが好き。...」  │
├─────────────────────────────────┤
│ [直近6ターンの生の会話履歴]     │  ← ここだけフル形式
└─────────────────────────────────┘
```

### 実装内容

**履歴上限の設定**:
- 直近 **6往復（user+assistant 計12メッセージ）** を上限とする
- 設定値は `config.yaml` に `max_history_turns: 6` として追記し、動的に変更できるようにする

**コンパクション処理**:
- 上限を超えたターンが発生したとき、超過分の古いメッセージをLLMに渡して要約する
- 要約プロンプト例:
  ```
  以下の会話履歴を3〜5文の日本語で簡潔に要約してください。
  重要な出来事・ユーザーの好み・決定事項を優先してください。
  ```
- 要約結果を `memory/compaction_YYYYMMDD.txt`（ワークスペース内）に保存し、次の呼び出し時にシステムプロンプトの直後に挿入する
- コンパクションはレスポンスをブロックしないよう非同期で実施すること（`threading` または `asyncio`）

**注意**: OpenClawの履歴管理がサーバー側にある場合は、ローカル側で別途履歴を管理するラッパーを実装してください。実装方針は調査結果を踏まえてAntigravityが判断し、完了報告書に記載すること。

## タスク A-2: プロンプトキャッシュ

### Anthropic Prompt Cachingの仕様

| 項目 | 内容 |
|---|---|
| 対応モデル | claude-haiku-4-5-20251001 ✅ |
| キャッシュ有効時間 | 5分（ephemeral） |
| 読み込みコスト | 通常の10%（90%削減） |
| 書き込みコスト | 通常の125% |
| 最小キャッシュトークン数 | 1,024トークン |
| 設定方法 | メッセージの `content` ブロックに `cache_control` を付与 |

### 実装内容

システムプロンプト（静的な部分）の末尾にキャッシュ境界を設定します。

**Python SDK経由の場合の例**:
```python
system_prompt = [
    {
        "type": "text",
        "text": "<SOUL.mdの内容 + TOOLS.mdの内容>",
        "cache_control": {"type": "ephemeral"}  # ← ここで境界設定
    }
]
```

**設定の優先順位**:
1. OpenClawが直接Python SDKを使っている場合 → APIコール部分に直接追加
2. OpenClawがCLI経由の場合 → 設定ファイルやラッパースクリプトで対応（調査結果次第）

**ログ出力**:
- APIレスポンスの `usage.cache_read_input_tokens` と `usage.cache_creation_input_tokens` をログに出力してください
- これによりキャッシュヒット率を実測できます
