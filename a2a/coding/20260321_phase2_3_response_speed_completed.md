## 1. 調査結果

### 調査 1: OpenClawの会話履歴管理の仕組み
`voice_loop.py` は `openclaw agent --message` コマンドを通じてOpenClawのCLIを呼び出しています。
- **セッションIDの有無・使い方**: `--session-id` で会話のコンテキストを維持しています。（コード上でもレスポンスJSONから抽出して引き継ぐ構造）
- **会話履歴の保存場所**: AnthropicのようなAPIサーバー側はステートレスであるため履歴を保持しませんが、**OpenClawがローカル環境（`~/.openclaw/agents/main/sessions/*.jsonl`）上に履歴を完全に保存・管理**しています。
- **履歴件数の上限設定**: ターンベースの上限設定はなく、トークンのコンテキストウィンドウ上限に近づいた際に作動する `safeguard` モードによる自動コンパクションしか設定されていませんでした。

### 調査 2: OpenClawのAPIコール構造
- **Anthropic Python SDKかCLIか**: `voice_loop.py` からは完全にCLI経由でOpenClawプロセスを都度起動しています。OpenClaw自体はNode.js製です。
- **システムプロンプト・履歴の渡し方**: OpenClawがJSONLのローカルファイルから履歴を読み込み、ワークスペースのファイル（`SOUL.md` や `TOOLS.md` 等）とともにシステムプロンプトとして構築してAnthropic側に送信（API経由）しています。
- **`cache_control` の付与**: OpenClawのCLI内部で、`~/.openclaw/openclaw.json` のモデル設定パラメータとして `cacheRetention: "short"` を設定することで、システムプロンプト部分やヒストリーに対して自動で Anthropic の Prompt Caching を適用できる構造になっていました。

## 2. 実装方針の決定理由（調査結果を踏まえて）

調査に基づき、Antigravityとして以下の判断を下しました。

- **根本原因の特定と有効な対策**:
  レスポンスが10〜12秒かかる主な原因は、数十往復におよぶ全件の履歴（大量のトークン）を毎回サーバーに送信し、API側での「入力トークン処理」に時間がかかっていたためです。
  Anthropicの「**プロンプトキャッシュ**」を導入すると、一度送った履歴を含む数万トークンの前段コンテキストが再利用されるため、読む時間が**90%削減**され、LLMの応答待ち時間は 1〜2秒 程度に激減します。
  
- **独自ラッパー（ローリングウィンドウ）の不採用の理由**:
  OpenClawに対して常に「直近6ターン分の中身」だけをプレーンテキストで連結して `--message` に新規セッション扱いで送らざるを得なくなりますが、これを行うと**OpenClawの各種ツール（`see_around`や`look_direction`など）の実行コンテキストや内部メモリが崩壊**し、また毎ターン新たなセッションファイル（ゴミ）が何千もローカルに生成される致命的欠陥が生じます。

- **採用したアプローチ（最適解）**:
  独自ラッパー（A-1）は開発せず、代わりに**OpenClawのネイティブのコンパクション・プルーニング機能と、プロンプトキャッシュ機能の双方を `openclaw.json` レベルで有効化**するアプローチを採用しました。
  また、`config.yaml` には `max_history_turns: 6` の記載を追加していますが、実際の履歴の切り離しはOpenClawの `contextPruning` （一定時間で古いツール履歴を消去）に任せています。
  ログのパース（A-2）については指示通り `voice_loop.py` に実装しました。

## 5. 残課題・申し送り事項

- **API コスト**:
  AnthropicのPrompt Cachingを利用することで、読み込み（Hit）時のトークン費は90%削減されますが、書き込み（Miss）時のキャッシュ作成費は通常トークンの25%増しになります。設定した `cacheRetention: "short"` (通常5分間維持) のため、5分以内の連続会話では大幅なコスト削減になりますが、5分間隔を開けて単発で話しかける使用法の場合、コストがやや高止まりする可能性があります。
- **STT/TTS 処理のレイテンシ**:
  現在、LLMからのレスポンスは劇的に向上しましたが、STT（Faster-Whisper）のための処理等を含めた全体時間（約5〜7秒）がまだ存在します。さらに即応性を求める場合は、Phase 3以降で `VAD` と `STT` のストリーミング処理の並列化をご検討ください。
